# Codebase Analysis: Claude Code Bridge System

## 1) Overall Project Understanding (Purpose and High-Level Functionality)

The system is an **all-in-one integration layer** designed to bridge the capabilities of the Claude Code AI environment with external, stateful devices, specifically an ESP32-based hardware robot.

**Core Purpose:** To extend Claude Code's functionality and observability by allowing external applications (dashboards, companion apps, custom services) to interact with a live AI session in real-time.

**High-Level Functionality:**

1.  **External Control (Inbound):** External clients (via HTTP REST or WebSocket) can inject messages directly into a running Claude Code session, as if the user typed them.
2.  **State Observation (Outbound):** The bridge broadcasts critical session events from Claude Code (e.g., AI replies, hook events, and tool-use requests) to connected external clients.
3.  **Permission Management:** This is the primary use case. External clients can approve or deny tool-use requests generated by the AI remotely, fulfilling the need for a unified interface for AI interaction, even in a detached environment.
4.  **Embedded Status:** The companion ESP32 firmware provides a real-time, state-driven dashboard (OLED) displaying the current status of the Claude session derived entirely from the communication bridge.

## 2) Code Layout (Detailed Directory Structure and Module Functions)

The system is split into three primary components, each with a distinct role:

### `plugin/` (Node.js Bridge Server)
This directory contains the installable Claude Code plugin, which serves as the central process.

*   `src/index.ts`: The core entry point, responsible for initializing and wiring together all I/O layers and services.
*   `src/mcp.ts`: Implements the Model Context Protocol (MCP) server. This module listens on `stdio` and handles Claude Code-specific features like the `reply` tool and crucial permission request handling.
*   `src/http.ts`: Manages the REST API layer (`/api/*`), providing endpoints for querying session state and submitting injections/verdicts remotely. Runs on port 8787.
*   `src/ws.ts`: Manages the WebSocket hub, providing a live, low-latency broadcast channel for continuous event streams to client applications. Runs on port 8787.
*   `src/bus.ts`: The central `EventEmitter` bus. It decouples the HTTP, WS, and MCP listeners, allowing events (e.g., `inbound_message`, `permission_request`) to flow seamlessly between I/O layers.
*   `src/state.ts`: The in-memory store for the conversation log (`Map`) and tracking pending permission requests.
*   `src/auth.ts`: Handles the implementation of shared-secret token validation for both HTTP and WS clients.
*   `src/types.ts`: Defines the shared TypeScript contract and message structures.
*   `plugin/.mcp.json`: Registers the bridge process with Claude Code, defining its capabilities and entry command.

### `robot_experiment/` (ESP32 Firmware)
This directory holds the Arduino sketch for the companion device, focused on high responsiveness and state-driven display.

*   `robot_experiment.ino`: The main entry point (`setup`/`loop`), which initializes and wires all other modules.
*   `ClaudeEvents.{h,cpp}`: The **core state management module**. It holds the complete, polled state of the Claude session (`working`, `session_id`, `last_tool`, etc.) and dispatches events to user-defined callbacks.
*   `BridgeClient.{h,cpp}`: Implements the WebSocket client, facilitating the connection to the central Node.js bridge server. It handles reconnection and JSON decoding of incoming frames.
*   `Display.{h,cpp}`: The output renderer for the 128×32 OLED. It is *fully state-driven*, meaning it never draws imperatively but reads the current state from `ClaudeEvents` to determine what needs to be rendered (e.g., spinner, tool label, summary).
*   `Motion.{h,cpp}`: Controls the servo motor. It handles keyframe playback for attention patterns (waggle) and slewing motions (`playJog`). This module is responsible for implementing smooth transitions (smoothstep easing) during operation.
*   `AmbientMotion.{h,cpp}`: The intelligent motion layer. It observes state transitions in `ClaudeEvents` to trigger meaningful behavior:
    *   *PreToolUse:* Fires a subtle jog movement when the AI is preparing a tool.
    *   *Working:* Enables a slow, sinusoidal oscillation mode when the AI is actively thinking (no concrete tool is firing).
*   `AttractScheduler.{h,cpp}`: A state machine that manages the idle behavioral cycle. When the system is quiet, this scheduler autonomously triggers attention patterns (`playWaggle`) at predetermined intervals.
*   `ToolFormat.{h,cpp}`: Translates complex AI tool metadata (e.g., `glob`, `bash`) into simple, display-friendly labels and one-line descriptions for the OLED.

## 3) Architecture (Deep Dive into Core Systems)

The architecture is characterized by a separation of concerns between the centralized bridge (Internet/AI-facing) and the embedded system (Hardware-facing), all linked by a highly structured, asynchronous event model.

### 🌐 The Node/MCP Communication Protocol (Central Bridge)

The Node.js bridge functions as a high-throughput, single-process hub, mandated to be minimal.

*   **Transport Layers:** The bridge manages three distinct communication transports:
    1.  **Stdio (MCP):** The direct communication link to Claude Code. This is the *primary* interface for receiving commands, sending responses, and receiving permission prompts. **Strict stdout hygiene** is required: only MCP JSON frames can touch stdout; all logging goes to `stderr`.
    2.  **HTTP REST API:** Used for synchronous, request-response interactions (e.g., `POST /api/messages`, `GET /api/state`). This is designed for durable external clients.
    3.  **WebSocket Hub:** Used for asynchronous, persistent, real-time event streaming (e.g., `outbound_reply`, `permission_request`). This is designed for low-latency UI updates.
*   **The Event Bus (`bus.ts`):** This mechanism is the heart of the plugin. All I/O activities—whether an HTTP POST, a WS message, or an MCP notification—are normalized into a generic event and broadcast via the bus. This means the HTTP layer doesn't need to know about the WS layer, and vice versa.
*   **Permission Gating:** When Claude Code sends a `notifications/claude/channel/permission_request` (e.g., requesting `ls`), the bridge captures it via `mcp.setNotificationHandler`. Instead of blocking, it stores this request in `state.ts` and emits a `permission_request` event on the bus. The external client must then explicitly call a remote endpoint (`POST /api/permissions/:id`) or send a WS verdict (`permission_verdict`) to resolve it.

### 🤖 ESP32 Firmware Loop & State Management (Embedded System)

The embedded system utilizes a **polled-state primary architecture**, avoiding imperative or callback-heavy design to ensure predictability and stability.

*   **The `ClaudeEvents` Latch:** The system's primary state is managed by `ClaudeEvents`. Crucially, during an active AI session, a "session latch" is held. This ensures that only events belonging to the currently active, relevant session are processed, preventing stale or unrelated tool-use signals from interfering with continuous operation.
*   **Decoupled Loop:** The `loop()` is not a single sequence of actions. It is a cooperative, periodic tick cycle where independent modules (`Display::tick`, `Motion::tick`, `BridgeClient::tick`, `AmbientMotion::tick`) run their logic based on the current state. This guarantees that the servo motion, the display refresh, and network health checks all operate concurrently and deterministically.
*   **Non-Invasive Behavior:** Modules like `AmbientMotion` and `AttractScheduler` do not register for event callbacks; instead, they *poll* `ClaudeEvents::state()`. If `working` state is set, `AmbientMotion` observes the subtle transition edges (like `PreToolUse`) to trigger physical responses without interfering with user-defined logic that might register a single event handler for that slot.

### 🔗 Communication Bridge Enablement (How it all fits)

The bridge is the necessary intermediary that translates the abstract, high-level operations of the AI into concrete, low-level, physical commands:

1.  **AI Intent $\rightarrow$ Bridge Event:** Claude Code decides to use a tool (e.g., `glob`). The MCP layer receives the `permission_request` $\rightarrow$ Node bridge stores it and broadcasts a JSON packet: `{"type":"permission_request", "request_id": "abcde", ...}` (via WS/HTTP).
2.  **Bridge Event $\rightarrow$ Firmware State:** The `BridgeClient` in the ESP32 receives this JSON packet $\rightarrow$ It dispatches an `permission_request` event to `ClaudeEvents` $\rightarrow$ `ClaudeEvents` updates the `pending_permission` state.
3.  **Firmware State $\rightarrow$ Motion/Display:** The `Display` module reads the `pending_permission` state $\rightarrow$ It renders a visible on-screen prompt: "ALLOW? `glob`...". Simultaneously, the `AmbientMotion` module observes the `tool_name` state change and triggers `Motion::playJog()` to provide physical feedback (a slight movement) that the AI is about to act.
4.  **External Resolution $\rightarrow$ AI:** An external client approves the prompt $\rightarrow$ The client sends a `permission_verdict` (WS/HTTP) $\rightarrow$ The bridge relays this verdict back to Claude Code via an MCP notification $\rightarrow$ Claude Code continues the session.
